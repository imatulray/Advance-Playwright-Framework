Execution Flow

A typical run moves from command to configuration, into tests, through reusable framework layers, and finally into reports.

1. Command npm test or another script in package.json starts Playwright.
2. Runner config playwright.config.ts sets testDir, browser project, retries, artifacts, and reporters.
3. Test files src/tests/*.spec.ts describe scenarios and assertions for login, product, checkout, and CURA flows.
4. Fixtures src/fixtures injects page objects, modules, and authenticated browser contexts.
5. Framework layers Modules call Pages, Pages use Playwright locators, and API classes call request helpers.
6. Output Playwright HTML, JSON results, screenshots, traces, video, and TTA live reports are written to output folders.

The Main Layers

The project separates what a test wants from how the browser or API performs it.

Tests

src/tests

Own the scenario names, tags, test steps, and final expectations. They should read like a clear checklist of behavior.

Modules

src/modules

Own business workflows such as logging in, adding a product to cart, buying now, and checkout actions.

Pages

src/pages

Own locators, single UI interactions, and page-level assertions. BasePage.ts centralizes common page helpers.

Support

fixtures, testdata, config, utils, api

Wire the layers together, keep data typed, read environment values, support API checks, and generate reports.

Example: login.spec.ts relationship test -> module -> page 
// src/tests/login.spec.ts
import { test, expect } from '../fixtures';
import { LoginModule } from '../modules/LoginModule';
import usersData from '../testdata/users.json';

test('should login successfully', async ({ page }) => {
  const loginModule = new LoginModule(page);
  await loginModule.doLogin(username, password);
  await loginModule.verifyLoggedIn();
  expect(page.url()).toContain('/home');
});

Folder Roles

Each folder has a fairly narrow job, which keeps new tests easier to place and existing tests easier to debug.

How To Read A Test

When you open any spec file, follow the imports first. They show which framework pieces the test relies on.

UI example

login.spec.ts imports LoginModule, users.json, and typed data from testdata/types.ts. The module creates LoginPage and HomePage, then calls page-level actions like filling fields, clicking buttons, and checking the user avatar.

API example

ProductApi.ts defines product response types and uses ApiHelper to send HTTP requests. That keeps API assertions separate from page locators and browser workflows.

Where New Code Goes

A quick placement guide for extending the framework without mixing responsibilities.

Need Put it here Why
New user-facing scenario src/tests/*.spec.ts The test file should name the behavior and hold the final assertions.
Reusable multi-step browser flow src/modules Modules keep tests readable by hiding repeated business workflows.
New locator or one-click action src/pages Page objects are the source of truth for how a specific screen is controlled.
Reusable logged-in state or injected object src/fixtures Fixtures share setup without duplicating it in every spec.
Static input data src/testdata Data files keep test values separate from test logic and allow TypeScript typing.
Environment setting .env, .env.example, src/config/index.ts Config centralizes URLs, credentials, timeouts, retry counts, and log level.
Run output or debugging artifact test-results, playwright-report, tta-report These are generated folders produced by Playwright and the custom reporter.
